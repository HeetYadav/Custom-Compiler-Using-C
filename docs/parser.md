# Stage 2: Parser

**Files:** `src/parser.c`, `include/parser.h`  
**Build flag:** `-DPARSER_STANDALONE`  
**Input:** a `.hehee` source file path, optional output `.dot` path  
**Output:** an `AstNode*` tree in memory + a Graphviz DOT file on disk

---

## What It Does

The parser takes the flat token stream from the lexer and structures it into an Abstract Syntax Tree (AST). The AST is the shared data format that both the semantic analyser and the ICG operate on, so getting this representation right is the central challenge of the front-end.

The parser also doubles as an AST visualizer: after building the tree it serializes it to a Graphviz DOT file, giving a direct graphical view of how the compiler understood the program structure.

---

## Grammar and Strategy

The parser implements a hand-written **recursive-descent parser** — each production rule in the grammar maps directly to a C function. The grammar it handles is formally documented in `include/parser.h`; in plain terms it covers:

- Top-level function and variable declarations
- Full block statements with nested scopes
- `if`/`else`, `while`, `for`, `return`, `break`, `continue`
- All expression levels from assignment down to primary (literals, identifiers, parenthesized expressions)
- Function calls (including multi-argument)
- Prefix and postfix `++`/`--`

This is an LL(1) grammar with one token of lookahead, which is exactly what the `Parser.current` field holds.

---

## Key Parsing Functions

Each level of the expression grammar is a function that calls the next-higher-precedence level:

```
parse_expr → parse_assign → parse_or → parse_and → parse_eq
          → parse_rel → parse_add → parse_mul → parse_unary
          → parse_postfix → parse_primary
```

The repetitive binary-operator levels (`parse_or`, `parse_and`, etc.) are generated by a single macro `BINOP_LEVEL`:

```c
#define BINOP_LEVEL(fn_name, next_fn, condition)                        \
static AstNode *fn_name(Parser *p) {                                    \
    AstNode *lhs = next_fn(p);                                          \
    while (condition) {                                                  \
        /* consume op, recurse right, create NODE_BINOP */              \
    }                                                                    \
    return lhs;                                                         \
}
```

This keeps the repetitive left-associative binary levels compact without sacrificing readability.

## AST Node Design

Every node has the same fixed-size structure, regardless of what kind of construct it represents:

```c
typedef struct AstNode {
    NodeKind       kind;
    char           text[256];          // name, operator symbol, or type annotation
    int            line;
    int            num_children;
    struct AstNode *children[64];      // up to 64 child pointers
} AstNode;
```

The `text` field carries different meanings depending on `kind`:
- For `NODE_FUNC_DECL`: `"func  main  :  int"` (encodes name and return type)
- For `NODE_VAR_DECL`: `"var  x  :  int"` (encodes name and type)
- For `NODE_BINOP` / `NODE_UNOP`: the operator symbol (`"+"`, `"!="`, etc.)
- For `NODE_IDENT`: the identifier name
- For literals: the literal's text value

This encoding trades a bit of elegance for simplicity — the semantic analyser and ICG parse the `text` field back out using `strrchr` and `extract_name()` rather than storing them as separate fields.

---

## Error Recovery

Rather than aborting on the first syntax error, the parser tries to continue. When `expect()` fails (the required token is not present), it logs the error, increments `p->errors`, and returns a dummy token. If a declaration or statement cannot be parsed, the fallback inserts a `NODE_ERROR` node as a placeholder child and advances past the unexpected token. This lets the parser report multiple errors in a single pass.

---

## DOT Output

The `ast_to_dot()` function does a depth-first traversal and emits a Graphviz `digraph`. Every node gets a distinct color and shape based on its `NodeKind`:

| Kind | Shape | Color |
|------|-------|-------|
| `NODE_PROGRAM`, `NODE_FUNC_DECL` | box | blue/green tints |
| `NODE_IF`, `NODE_WHILE`, `NODE_FOR` | diamond | warm tints |
| `NODE_RETURN` | trapezium | purple tint |
| `NODE_CALL` | ellipse | cyan tint |
| Literals | box | yellow tint |
| `NODE_ERROR` | octagon | red |

Each node label shows the `text` field and the source line number. Rendering the DOT file with `dot -Tpng` gives an immediate visual sanity-check of the parse result.

## Example

For the `add` function in `tests/test.hehee`, the resulting AST fragment looks like:

```
NODE_PROGRAM
└── NODE_FUNC_DECL  "func  add  :  int"
    ├── NODE_BLOCK  "params"
    │   ├── NODE_PARAM  "int  a"
    │   └── NODE_PARAM  "int  b"
    └── NODE_BLOCK  "block"
        └── NODE_RETURN  "return"
            └── NODE_BINOP  "+"
                ├── NODE_IDENT  "a"
                └── NODE_IDENT  "b"
```
